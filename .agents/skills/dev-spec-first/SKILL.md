# dev-spec-first

**先写规格，再决定是否开发。**

这个 Skill 的核心规则只有一条：
**除非用户明确说"直接开发 / 直接改代码 / 应用修改"，否则默认只输出规格 prompt，绝不动代码。**

---

## 触发条件

当用户发来以下任意内容时，自动激活本 Skill：

- 截图 + 文件路径
- 自然语言需求描述（"帮我改…" / "我想让…" / "这个按钮需要…"）
- 参考页面 + 当前项目文件路径
- 接口线索 + 功能需求
- 多轮补充需求

---

## 默认模式：规格整理（Spec Mode）

**这是默认行为，优先级最高。**

收到需求后，你必须先完成以下动作，**不要直接读文件、不要直接改代码**：

### Step 1：需求分析

快速识别以下要素：

- **改什么**：功能/样式/逻辑/结构？
- **在哪里**：哪个文件、哪个区域、哪个组件？
- **参考物**：是否有参考 HTML / 设计稿 / 截图？
- **边界**：哪些区域绝对不能动？
- **复用要求**：哪些现有逻辑必须复用？

如果有明显的信息缺口（如缺少文件路径、缺少参考），**只问最关键的 1 个问题**，不要连续多问。

### Step 2：输出规格 Prompt

按以下结构输出，语言自动跟随用户输入语言：

```
## 开发 Prompt

### 任务描述
[一句话说清楚这次要做什么]

### 文件路径
- 目标文件：[路径]
- 参考文件：[路径，如果有]

### 改动范围
只允许修改：
1. [具体区域/节点/方法]
2. [...]

### 禁改区域
不允许修改：
1. [区域/组件/逻辑]
2. [...]

### 复用要求
必须复用：
1. [现有方法/数据/逻辑]
2. [...]

### 实现约束
- [技术栈限制，如 Vue2 禁用语法]
- [组件库限制]
- [命名规范]

### 输出要求
按以下顺序输出：
1. 识别到的当前节点
2. 需要替换的 template/代码片段
3. 需要替换/新增的样式片段
4. 如必须改 script，只给最小改动并说明原因
5. 明确列出"本次没有改动的区域"

### 验收标准
- [✅ 条件1]
- [✅ 条件2]
- [❌ 不应出现的情况]
```

### Step 3：询问执行意图

规格输出完毕后，在末尾附加一行：

> 📋 规格已就绪。如需执行，请说"**直接开发**"或"**应用修改**"。
> 如需要调整规格，直接告诉我。

---

## 模式切换：直接开发模式（Dev Mode）

**只有用户说出以下关键词之一，才进入此模式：**

- 直接开发
- 直接改代码
- 应用修改
- 应用补丁
- 改文件
- 开始写
- Execute / Apply / Implement（英文场景）

进入 Dev Mode 后，执行方式如下：

### 执行者选择

| 场景 | 执行者 |
|---|---|
| 默认（solo 模式） | 我（solo agent）直接读取文件并修改 |
| 用户要求团队模式 | 切换到 master crew 拆分任务，分发给 frontend/backend 等子角色 |
| 用户要求 ACP harness（Codex/Cursor 等） | 将规格 prompt 适配后发送给对应 harness |
| 用户要求手动应用 | 只输出可直接复制的代码片段，不动文件 |

### Dev Mode 行为准则

1. **先读文件，不猜结构** — 必须先读取真实文件再动手
2. **最小改动** — 只改规格里标注的区域
3. **不重写整页** — 禁止整文件覆盖，优先精确替换
4. **先落盘再回复** — 文件写完才回复结果
5. **改完列清单** — 输出"改了什么 / 没改什么"的对比清单

---

## ACP Harness 适配

当用户说"用 Codex 做 / 用 Cursor 做 / 用 Claude Code 做"时：

1. 将规格 prompt 转换为对应 harness 的标准输入格式
2. 通过 `sessions_spawn(runtime="acp")` 发送任务
3. 等待结果后汇报

---

## 多轮对话处理

- 用户补充新信息时，**更新规格，不重新全量输出**，只标注"更新了哪一项"
- 用户直接说"加一条约束"/"禁止改 XXX"时，立刻追加到禁改区域
- 规格确认后进入 Dev Mode，**不再重复输出规格**，直接执行

---

## 快速参考卡

```
用户输入 → 需求分析 → 输出规格 Prompt → 等待确认
                                          ↓
              用户说"直接开发"    →    进入 Dev Mode → 修改文件
              用户说"手动应用"    →    输出代码片段
              用户补充需求        →    更新规格
              用户转发给 harness  →    适配发送
```

---

## 注意事项

- **永远不要**在用户没说"直接开发"时主动修改文件
- **永远不要**在规格不完整时跳过澄清直接猜测开发
- 规格 prompt 要足够完整，让任何 coding agent 拿到后可以独立执行，不需要再问问题
- 如果需求涉及业务逻辑，规格里必须明确标注"复用现有 XXX 方法，不重写"
- **复杂任务必须先拆成小模块，再一步一步开发**
- 进入 Dev Mode 后，复杂需求默认按“小步骤串行执行”处理：**一次只做一个小步骤，每做完一步先汇报结果，再进入下一步**
- **不要把额外优化、顺手修复、附加防坑逻辑混进当前步骤**；如有建议，先单独说明，等待用户确认
